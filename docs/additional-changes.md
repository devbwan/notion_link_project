# 추가 변경사항

## 페이지 내부의 하위 페이지 추출

### 현재 동작

현재 구현은 지정한 최상위 Notion 페이지의 본문 블록을 재귀적으로 읽는다. 일반 중첩 블록은 추출하지만 `child_page` 블록은 하위 페이지의 제목만 기록하며, 하위 페이지 본문은 별도로 조회하지 않는다.

### 변경 요구사항

페이지 내부에 포함된 하위 페이지도 문서로 인식하여 본문을 추출한다. 하위 페이지 안에 또 다른 하위 페이지가 있으면 동일한 방식으로 재귀 처리한다.

```text
메인 페이지
├─ 본문
├─ 하위 페이지 A
│  ├─ 본문
│  └─ 하위 페이지 B
└─ 하위 페이지 C
```

### 구현 방향

1. 상위 페이지의 블록을 조회한다.
2. `child_page` 블록을 발견하면 해당 블록 ID를 하위 페이지 ID로 사용한다.
3. 하위 페이지의 메타데이터와 자식 블록을 별도로 조회한다.
4. 발견한 모든 하위 페이지에 같은 처리를 재귀적으로 적용한다.
5. 이미 방문한 페이지 ID를 기록하여 중복 조회와 순환 처리를 방지한다.
6. 비정상적으로 깊은 구조에 대비해 최대 탐색 깊이를 설정한다.

각 하위 페이지도 Notion integration에 공유되어 있어야 한다. 접근 권한이 없는 페이지는 다른 페이지의 처리를 중단하지 않고 오류 또는 경고로 기록한다.

### 출력 정책

하위 페이지는 상위 문서에 모두 합치지 않고 페이지별 파일로 저장한다.

```text
output/notion/<main-page-id>.md
output/notion/<child-page-a-id>.md
output/notion/<child-page-b-id>.md
```

페이지별 파일 분리는 다음 이점을 갖는다.

- 페이지 수정 시 해당 파일만 갱신 가능
- 동일한 하위 페이지를 여러 위치에서 참조해도 중복 파일 방지 가능
- 깊은 문서 구조에서도 파일 크기와 처리 범위를 제한 가능
- 페이지 ID 기반 상태 저장 및 멱등성 정책을 그대로 적용 가능

상위 페이지 파일의 `child_page` 위치에는 하위 페이지 제목과 생성된 로컬 파일의 상대 링크를 기록한다.

### 완료 기준

- 한 단계 하위 페이지의 제목과 본문이 별도 파일로 생성된다.
- 여러 단계로 중첩된 하위 페이지가 모두 처리된다.
- 상위 페이지에 하위 페이지 파일의 상대 링크가 생성된다.
- 동일한 페이지 ID는 한 번만 추출된다.
- 최대 깊이 초과와 접근 권한 오류가 안전하게 보고된다.
- 일부 하위 페이지가 실패해도 접근 가능한 다른 페이지는 계속 처리된다.
- 반복 실행 시 변경되지 않은 페이지 파일은 다시 쓰지 않는다.
